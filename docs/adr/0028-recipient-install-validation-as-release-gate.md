# ADR 0028: Recipient-install validation as a release-time gate

- **Status:** Accepted
- **Date:** 2026-05-07

## Context

Through v6.10.5, the team's release-time gate stack — pytest, ruff, the standalone security probe, CLI smoke, `ci-gate-runner`, `mcp-protocol-auditor`, `cue-card-rehearsal-auditor`, `reproducibility-provenance-auditor`, `phi-irb-risk-reviewer`, `coverage-criticality-mapper`, `red-team-reviewer` — has been the team's mechanism for asking *"is this release shippable?"* Every gate runs against the developer's machine state: the source tree on disk, the dev's Python install, the dev's locale, the dev's prior `~/.tailor/` state, the dev's Claude Desktop install variant.

That gate stack does not test the **seam between the wheel artifact and a stranger's machine**. The four consecutive patch releases v6.10.1 → v6.10.4 are the team's evidence that this gap is load-bearing:

1. **v6.10.1 (cp1252 glyphs)** — `→` in `cmd_status` crashed in cp1252 PowerShell on a Windows recipient. Dev machine had UTF-8 locale; failure invisible to every host-side gate. Caught only when the recipient (dad) ran `tailor tour` cold on Win 11 PowerShell on 2026-05-06.
2. **v6.10.2 (SetupHelpLayer + RECIPIENT_README)** — a v6.9.x failed-tour install routed the recipient via web-Claude into a degraded `serve` state with no demo tools. Closure required a structural framework-tier addition; the failure mode itself was undetectable by the dev gate stack because it required the recipient's machine ending up in a partially-installed state the dev's machine never reaches.
3. **v6.10.3 (sibling-cleanup)** — multi-`biosensor-*` entries in the recipient's `claude_desktop_config.json` after `tour --force`. Caused by web-Claude-mediated debugging on the recipient's prior failed install leaving a stale entry. Failure mode required the recipient's specific config-file history; uncatchable on a dev machine that always starts from a clean config.
4. **v6.10.4 (dual-path Claude Desktop config)** — Microsoft Store / Classic UWP-sandbox mismatch on Windows. The Store version of Claude Desktop reads from a per-package sandbox path the framework never wrote to. Dev's Claude Desktop was the Classic install; failure invisible until a Store-installed recipient hit it.

The pattern across v6.10.1–v6.10.4 is structural: every failure required *recipient state the dev machine cannot reach by construction*. cp1252 locale, prior-failed-install state, an alternate Claude Desktop install variant — each one recipient-specific, each one undetectable by gates that test against the dev tree. The team's existing host-side coverage compounds with the project; the recipient-install seam does not.

[ADR 0026](0026-claude-desktop-config-dual-path.md) named the gap explicitly in its test-plan checkbox: `[ ] Manual: tour --force on a Windows machine with Store Claude Desktop installed (validates the UWP path write — automated tests mock the glob)`. Manual-test-plan checkboxes are the durable record of where the gate stack stops.

The structural question this ADR answers: *given that the recipient-install seam has produced four consecutive ship-blocker patches in nine days that the existing gate stack cannot reach by construction, what closes the seam at PR time?*

## Decision

A new specialist agent, **`recipient-install-validator`**, is promoted to the team's roster as a release-time gate. The agent provisions a clean Windows 11 base box via VirtualBox + Vagrant, installs the freshly-built wheel via the documented recipient command, runs `tailor tour`, validates per-path Claude Desktop config (per ADR 0026), exercises `tailor demo` (per ADR 0027), and runs wheel-install-dependent pytest in-guest.

The agent is **mandatory + file-touched-gated**: it fires when any of `tour.py`, `pilot.py`, `__main__.py`, `wizard.py`, `pyproject.toml` package-data globs, or `_fixtures/**` are modified in a release branch. The trigger pattern matches `cue-card-rehearsal-auditor` per [ADR 0025](0025-cue-card-rehearsal-as-release-gate.md) — files-changed determines whether the gate fires; when it fires, `release-shipper` blocks on its verdict.

Concrete mechanism:

- **Layered coverage (L1 + L2).** L1 is the install ritual + serve smoke: 8 ordered assertions inside the Win 11 guest covering `pip install`, `tailor tour` (with v6.10.4 exit-code semantics — exit 0 means *"at least one path written,"* not *"everything worked"*; partial-success is `WARN`, not `PASS`), per-path Claude Desktop config inspection with the v6.10.4 cross-path-identity invariant, scaffold integrity, `tailor demo` with the v6.10.5 / [ADR 0027](0027-demo-as-researcher-first-look.md) "HIP Lab not Strava" framing assertion, `tailor status` recovery-framing assertions, and `tailor serve` startup. L2 is in-guest pytest of wheel-install-dependent test files (`tests/test_serve_mcp_protocol.py`, `tests/test_demo_runner.py`) — file selection deliberate, mocked unit tests deferred to `ci-gate-runner`'s host pass.
- **Hypervisor stack: VirtualBox + Vagrant on the boss's Win 11 Home host.** Empirically validated 2026-05-07: Linux smoke-test boot in ~3 min; Win 11 (Hyper-V emulation backend) first-boot in ~25 min once the Vagrantfile's `boot_timeout` is set to 1800s. The default Vagrant 5-minute timeout is insufficient on hosts with `VirtualMachinePlatform` enabled; the agent's canonical Vagrantfile encodes the empirically-derived value.
- **Snapshot-revert before each run.** The base image is `gusztavvargadr/windows-11` (Win 11 Enterprise — Microsoft does not publish a Home Vagrant box; edition differences do not sit on the recipient install path) with no Claude Desktop pre-installed. Vagrant snapshot revert is the load-bearing mechanic for "fresh recipient" purity across runs — without it, the guest accumulates state and the v6.10.x-class failure shape recurs.
- **Accepted v1 coverage gap (option α).** The base image has no Claude Desktop installed; the v6.10.4 dual-write code path therefore is not exercised on a real Windows guest in v1. Mocked unit tests in `tests/test_dual_path_registration.py` cover that path on the host pass. The v2 escalation path — fake-`Packages\Claude_<suffix>\` directory pre-creation in the base image (option β) — is named explicitly in the agent's prompt; it closes ADR 0026's manual-test-plan checkbox without requiring a real Microsoft Store install. Option γ (install real Claude Desktop variants in the base image) is parked for v3 with the maintenance-cost argument named.
- **Bash + Read + Write + Edit + Grep + Glob tools.** The agent shells out to `vagrant`, `VBoxManage`, and `pytest`; reads guest stdout/stderr; writes the canonical Vagrantfile to the temp project dir; edits the Vagrantfile when the boot_timeout needs tuning per host. `model: opus` because the agent makes natural-language failure-diagnosis judgments (parsing tour output, classifying unexpected stderr) that benefit from stronger reasoning, even though the per-step assertions themselves are mechanical.
- **The agent's report format mirrors `mcp-protocol-auditor`'s PASS/FAIL/GAP shape**, extended with `WARN` for the v6.10.4 partial-write surface (`Tour scaffolded with N of M ...` exit-0 outcomes that are not failures but are also not full successes). Aggregate verdict: `RECIPIENT-INSTALL OK` / `RECIPIENT-INSTALL WARNINGS` / `RECIPIENT-INSTALL BROKEN`.

The rule, plain English: the team has been treating recipient-install failures as discoverable post-release because no gate caught them at PR time. This ADR closes that loop — recipient-install failures are now a release-blocker the same way wire-protocol regressions and schema-prompt mismatches already are.

### Promotion criteria mapping (per ADR 0011)

| Criterion | Where it lands |
|---|---|
| Structural argument | Host-side gates run against dev-machine state by construction; no agent owns the wheel-to-stranger's-machine seam. v6.10.1–v6.10.4 are the patches this seam produced — four ship-blocker bugs in nine days, all from the same uncovered surface. |
| Severity | Recipient-install failures are user-visible and blocking. Recovery requires manual debugging the framework's documentation does not anticipate; the recipient (dad's actual case) ended up routing through web-Claude into a degraded state with no biosensor tools, requiring three commands to recover from a single bug. |
| Per-agent maintenance estimate | ~4–8 hours/quarter (base image rebuilds when Python / Claude Desktop drift; Windows Update breaking the snapshot; Vagrant box version drift; agent prompt updates per release that triggers the gate) plus ~30–120 min per triggered release. Highest on the roster — accepted because the gap it closes is correspondingly the most expensive uncovered failure shape. |
| Frequency-based fallback (3+ uses) | Trivially passes — the surface produced four patches in nine days. Not the load-bearing argument; the structural argument carries the case. |

## Consequences

### Positive

- **The recipient-install seam is covered at PR time.** Four-patch failure history names the gap; the agent closes it. Future v6.10.x-class failures surface in the agent's verdict before tagging, not in dad's debugging session afterward.
- **The team's gate-stack architecture becomes coherent.** `ci-gate-runner` runs against the dev tree on the host; `recipient-install-validator` runs against the wheel-installed package on a clean guest. They compose at `release-shipper`: host gate first, guest gate second when the trigger fires. The split is now first-class rather than an implicit accident of which gates happened to exist. Mocked unit tests stay on the host (fast feedback); wheel-install-dependent tests run in-guest (slower but load-bearing).
- **The v6.10.4 manual-test-plan checkbox has a closure path.** ADR 0026's `[ ] Manual: tour --force on a Windows machine with Store Claude Desktop installed` is closed by this agent's v2 (option β — fake-Packages-dir pre-creation). The path forward is named, not deferred indefinitely.
- **The team's release ritual gains a recipient-perspective backstop.** Same shape as `cue-card-rehearsal-auditor` (ADR 0025) closing the schema-vs-prompt seam; this agent closes the wheel-vs-machine seam. The two share the file-touched-gated mandatory-pre-tag trigger pattern, making the team's gate-composition rules predictable.
- **The Vagrantfile's `boot_timeout = 1800` encodes a hard-won empirical lesson.** The 2026-05-07 first attempt timed out at the Vagrant default 300s; the second attempt with 1800s succeeded. That number is now documented in the agent's prompt rather than rediscovered every time someone wonders why their Win 11 box doesn't boot.

### Negative

- **The agent's per-run cost is the highest on the roster.** ~3–5 min on a warm snapshot revert; ~25 min on cold first-boot. File-touched-gating mitigates by not firing on releases that don't touch the install path; releases that do touch it pay the ~5 min routinely.
- **Maintenance burden is real.** Base image needs periodic rebuilds when Python minor versions drift or Claude Desktop ships breaking changes to its config schema. Vagrant box version drift requires periodic refresh. The agent prompt itself needs updates when the install ritual changes shape (as in v6.10.4). Estimated 4–8 hours per quarter, plus 30–120 min per triggered release. The team accepts this cost because the gap it closes is the most expensive uncovered failure shape on the roster.
- **VirtualBox runs in Hyper-V emulation mode on Windows 11 Home hosts** with `VirtualMachinePlatform` enabled (the modal recipient state and the boss's host state). The slow path is 2–4× slower than raw VT-x for CPU-bound boot work; the agent's Vagrantfile encodes a 30-min `boot_timeout` to survive this. Hosts without `VirtualMachinePlatform` enabled run faster but the agent's tuning is overly generous in that case (acceptable — false-fast is not a failure mode).
- **Win 11 Enterprise (the box) ≠ Win 11 Home (the modal recipient).** Edition differences (Group Policy editor, BitLocker by default) do not sit on the recipient install path; cp1252 default encoding, PowerShell behavior, file system layout, and Claude Desktop install path are identical. The mismatch is not load-bearing but is surfaced explicitly so the asymmetry is durable rather than implicit.
- **v1 base image has no Claude Desktop installed.** The dual-write code path from ADR 0026 is therefore not exercised on a real Windows guest in v1. Mocked unit tests in `tests/test_dual_path_registration.py` cover this on the host pass. v2 (option β) closes the gap on the guest via fake-Packages-dir pre-creation; named in the agent's prompt as an explicit escalation path.
- **The agent is bound to the boss's host.** v1 runs only where VirtualBox + Vagrant are installed and `gusztavvargadr/windows-11` is cached. A future contributor running the gate would have to provision the same stack — not a one-line install. Cloud-VM as v2 escape hatch is named in the agent's description as the portability path; not implemented in v1 because a hobby project with one host doesn't yet need it.

### Neutral

- **The agent is bound to Windows 11 by design.** macOS and Linux recipient surfaces are out of scope for v1. Mac App Store Claude Desktop variant, when Anthropic ships it, becomes a v3+ concern (named in [ADR 0026](0026-claude-desktop-config-dual-path.md) § Negative consequences and inherited here).
- **`release-shipper.md` is unchanged.** The trigger is encoded in the agent's own description; the main session orchestrates spawning during release prep. Same pattern as `cue-card-rehearsal-auditor` (ADR 0025). Triggers live next to the work, not in a central dispatcher.
- **The deferred roster in [docs/design/operating-model.md](../design/operating-model.md) gains a v2-named entry** for the option-β escalation path.
- **The agent's `model: opus` choice mirrors `cue-card-rehearsal-auditor`** rather than `mcp-protocol-auditor`'s `model: sonnet`. The reasoning surface is closer to cue-card-rehearsal — natural-language failure diagnosis on unexpected stderr — than to mcp-protocol-auditor's purely-mechanical wire-level pass-fail.

## Alternatives considered

**Cloud VM (Azure / AWS / Hetzner) instead of local VirtualBox.** Considered. Pros: zero local install pain, no maintenance burden of a base image on the boss's machine, releasable to other contributors who don't have VirtualBox set up. Cons: recurring cost (~$0.05–$0.50 per run × release cadence), credential management, network latency to provision (~1–3 min). Rejected for v1 on cost-vs-frequency grounds — for a hobby project with one host, recurring cloud cost outweighs the one-time setup pain. Named as the v2 escape hatch if local VirtualBox proves flaky on the host or the team grows to multiple contributors.

**Disable `VirtualMachinePlatform` on the host to get raw VT-x performance.** Considered. Disabling the feature would let VirtualBox run at full speed (2–4× faster). Cons: requires a reboot, removes Windows Sandbox / WSA / other Microsoft container features, and Windows Update may silently re-enable the feature without warning — at which point the agent silently degrades to slow-mode again. Rejected on the maintenance-fragility argument: a gate that quietly slows down across machine reboots is worse than a gate that's consistently slow.

**Install real Claude Desktop variants in the base image** (instead of running with no Claude Desktop installed in v1). Considered. Most accurate to the recipient experience. Rejected on maintenance grounds: re-imaging the base box every time Anthropic ships a Claude Desktop release; Microsoft Store unattended install in Vagrant is notoriously awkward; the marginal accuracy over option α (no Claude Desktop) for the v6.10.4 dual-write coverage gap is small relative to the ongoing cost. Named as v3 if v2's fake-Packages-dir approach proves insufficient.

**Drive Claude Desktop GUI automation inside the guest** (e.g. via `pywinauto`) to validate the full Claude-Desktop-mediated install path. Considered. Catches a class of failure where Claude Desktop's config schema silently changes and our merge no longer parses. Rejected: the failure class is rare and broadcasts (every MCP server in the wild breaks at the same moment); GUI automation tools are notoriously brittle; distinguishing "agent saw red because config is broken" from "agent saw red because Claude Desktop is still loading" is its own engineering problem; v6.x bug history has zero failures of this class. Not in v1, v2, or v3 — no evidence the gap exists.

**Drive a real Claude session against the in-guest MCP server** (cue-card prompts via API to confirm prompt-driven behavior end-to-end). Considered. Catches the v6.9.1 class (Claude infers wrong parameter from operator prose). Rejected — this is `cue-card-rehearsal-auditor`'s natural growth path (per [ADR 0025](0025-cue-card-rehearsal-as-release-gate.md)), not this agent's. Different orchestration shape (API keys, network, non-determinism), different trigger (cue-card / schema changes vs install-ritual changes), different cost class (API spend per run vs disk + time). Composing at `release-shipper` rather than overlapping is the cleaner separation.

**Skip the agent; rely on the v6.10.x quartet's accumulated documentation as the closure.** Rejected. Documentation enumerates known failure modes; it does not detect new ones. The argument-from-precedent — *"the team has now learned about cp1252, UWP sandbox, sibling-coexistence, dual-path, and partial-write surfaces"* — predicts that v6.11+ recipient-install failures will be different in shape from the ones the v6.10.x patches closed. Detecting *new* failure shapes is what the agent does that documentation cannot.

**Run the agent as a manual operator tool only (not a checked-in specialist).** Considered. The boss could spin up a clean Win 11 VM manually before any tour-touching release and run the install ritual by hand. Rejected on consistency grounds: the v6.10.x quartet shows that ad-hoc manual testing is exactly what didn't happen on the four patches that needed it. Promoting the gate from manual-when-remembered to mandatory-when-files-touched is the structural patch on "remember to test recipient install before shipping" — same shape as ADR 0010's adversarial-pairing argument that confirmation-shape dispatch produces confirmation-shape outputs.

**Snapshot-revert as the only cleanup mechanism.** Considered for the v1 spec and chosen — the v1 prompt's Phase 3 says explicitly "Do NOT `vagrant destroy`" and relies on `vagrant snapshot restore base` at the start of each run for cleanliness. Rejected for v6.11.x amendment after the v1 first-wild-run on 2026-05-07 produced an orphan running VM (155+ minutes of host CPU before manual cleanup). The mechanism's failure mode is structural: snapshot revert only fires *when the agent runs again*. A run interrupted mid-Phase-1 — by host reboot, by the dispatching session timing out, by the agent's own watcher silently parking — never reaches its successor's snapshot-revert step. Halt-on-exit (added in v6.11.x) closes this failure mode without modifying the v1 architecture (no destroy, snapshot preserved). See "v6.11.x amendments" below.

## v6.11.x amendments (post-first-wild-run)

The 2026-05-07 first wild invocation of `recipient-install-validator` (boss-requested capability test on clean main at v6.11.0) surfaced three structural gaps in the v1 prompt that the v6.11.0 promotion-pass dogfood did not catch. The dogfood ran successfully end-to-end; the gaps live on the *failure-handling* paths the dogfood did not exercise.

**Gap 1 — Halt-on-exit absent.** The v1 Phase 3 says `vagrant snapshot restore base` and explicitly says NOT to `vagrant destroy`, but says nothing about `vagrant halt`. On a successful run, snapshot-revert handles cleanliness. On a failed/interrupted/timed-out run, the guest VM stays running until the next invocation's snapshot-revert step — which may never come. Evidence: a 155-minute orphan VM (`biosensor-vagrant-win11-smoke_*`) from a prior session was already running on the host when the 2026-05-07 first-wild-run started. The agent (correctly) did not touch the orphan, but its own run also produced one when interrupted. **Patch (v6.11.x prompt):** halt-on-exit is mandatory on every exit path (PASS / WARN / FAIL / TIMEOUT-WATCHER-DEAD); snapshot-revert is preserved for the success path; destroy is still forbidden. Codified in the agent's "Safety rules" and "Phase 3 — Teardown" sections.

**Gap 2 — Watcher timeout enforcement unspecified.** The v1 Vagrantfile encodes `boot_timeout = 1800` but the v1 prompt does not constrain the agent's own watcher `timeout_ms` values. In the 2026-05-07 first wild run, the agent's three parallel watchers (vagrant-up, boot-success poller, boot-milestone poller) parked silently past the 30-minute boot envelope and continued for an additional 70+ minutes without firing a terminal event. The Monitor "silence is not success" failure mode is the structural cause: watchers that grep only for happy-path markers stay silent through hangs and crashloops. **Patch (v6.11.x prompt):** new "Watcher discipline" section requires `timeout_ms ≤ boot_timeout + 120s` and failure-signature coverage on every watcher. A watcher whose deadline expires without a terminal event yields the new `RECIPIENT-INSTALL TIMEOUT-WATCHER-DEAD` aggregate verdict, which still triggers halt-on-exit before reporting.

**Gap 3 — Mid-flight observability absent.** The v1 prompt's "Be terse" guidance and tabular final report give the dispatching session no signal during the audit's long tail (Phase 1 step 1's `pip install` of the wheel through `cryptography`/`lxml`-style C extensions on a Hyper-V-emulation guest, plus Phase 2 pytest). The 2026-05-07 first wild run was indistinguishable from a hang from the dispatching session's perspective. **Patch (v6.11.x prompt):** new "Progress emission" section requires `[validator] phase X step Y/N` heartbeat lines at every observable boundary, plus per-step `elapsed: <seconds>` columns in the final report.

These three patches do not change the agent's substantive correctness contract (the eight Phase 1 assertions, the cross-path-identity invariant, the HIP-Lab-not-Strava check, the in-guest pytest selection). They harden the failure-handling and observability paths — exactly the surface the v6.11.0 promotion-pass dogfood could not exercise because it ran the happy path successfully.

The operational-hardening amendments leave the v1 coverage gap (option α — no Claude Desktop in base image) and v2 escalation (option β — fake-Packages-dir pre-creation) as-named. They do, however, surface the cost evidence that motivates the policy refinement below.

## v6.11.x mandate refinement — opt-in, not mandatory

The original ADR 0028 wording said the agent is *"mandatory + file-touched-gated"* and that *"when it fires, `release-shipper` blocks on its verdict."* Both clauses turn out to be aspirational rather than implemented:

- **Aspirational mandate.** As of v6.11.0, `release-shipper.md` did not reference this agent at all. The gate fired only when the main session in release prep "remembered" to spawn it — convention, not enforcement. The same gap exists on ADRs 0016 / 0025, addressed there in parallel by the v6.11.x amendments to those ADRs (attestation enforcement at release-shipper pre-flight).
- **Cost evidence from the first wild run.** The 2026-05-07 capability test ran the agent for 101 minutes before manual interrupt; in normal usage on the boss's Win 11 Home host (Hyper-V emulation mode per § Negative consequences), a successful end-to-end run is 30–100 minutes. That cost is asymmetric with the other two file-touched gates: `cue-card-rehearsal-auditor` runs in seconds, `mcp-protocol-auditor` in minutes.

The cost-vs-frequency analysis the original ADR did not have:

- The bug history that justified the gate's promotion (v6.10.1 → v6.10.4) was clustered in a single nine-day window in early 2026-05. The install-path code has been quiet since.
- Forward-projecting to v6.12+, releases that meaningfully change the install path are likely 2–4× per year, not every release. The file-touched globs over-trigger — a typo fix in `tour.py` matches the trigger but has zero recipient-install risk.
- A 30–100 minute mandatory gate, fired on every install-path-touching release whether substantive or not, will routinely add an hour to release ceremony. Under release pressure, a heavyweight always-on gate pushes the team to silence-or-skip — which is the failure mode the original mandate was meant to close.

The v6.11.x mandate refinement: **the gate becomes boss-discretionary opt-in.** Mechanism per `release-shipper.md` § "Pre-tag gate composition":

- release-shipper inspects `git diff --name-only main...HEAD` against the trigger globs.
- If any glob matches, release-shipper SURFACES the recommendation in pre-flight output (and records it in the release commit body) — but does NOT refuse.
- The boss explicitly opts in via `--full-validate`. Then release-shipper spawns the agent inline (same shape as `ci-gate-runner` spawn at pre-flight step 3) and blocks on its verdict.
- Default is "do not fire." The boss judges when the install path has changed substantively enough to warrant the cost.

**Why opt-in rather than attestation-required.** The other two gates use attestation (boss runs the gate, attests verdict via `--gates-confirmed`) because their cost is low enough that running them every applicable release is cheap. This gate's cost is high enough that even attestation would create silence-or-skip pressure — the boss, knowing he has to run a 30–100 min gate to attest a verdict, will be tempted to skip it on a typo-fix release. Opt-in removes the implicit pressure: the recommendation is surfaced, the cost is named, the boss decides.

**What this gives up.** The "manual-when-remembered fails" structural argument from § Alternatives § "Run the agent as a manual operator tool only" softens to "manual-when-recommended-and-acknowledged." That is a real concession. The mitigating arguments:

1. The recommendation is now an active surface in release-shipper output, not a passive "should-have-remembered." A boss who declines is declining explicitly, not forgetting.
2. The release commit body records `## Pre-tag gates surfaced` — `recipient-install-validator: triggered, not opted-in` — making the decision auditable.
3. The cost asymmetry is real, and the original framing did not reckon with it. Opt-in is the policy that survives the actual cost; mandatory was the policy that would have eroded into silence on its own.

**What this does NOT give up.** The substantive correctness contract is unchanged: when the gate runs, it runs the full eight-step ritual with the v6.11.x operational hardening. The v1 coverage gap (option α) and v2 escalation (option β) are unchanged. The bug-history-based justification for the gate's existence is unchanged.

The original "mandatory + file-touched-gated" wording stands on the record; the v6.11.x amendment refines the policy to match the actual cost-vs-frequency profile the first wild run revealed.
